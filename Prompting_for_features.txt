## Security & implementation guardrails

The PRP must bake these into the relevant sections (Technical Requirements, Edge Cases, Acceptance Criteria) as concrete, testable requirements — not as a vague "handle securely" note. For each guardrail below, specify exactly what the endpoint must do.

**Input validation & sanitization**
- Every field has an explicit type, max length, and allowed-character rule (e.g. `title`: string, trimmed, 1–200 chars; `description`: string, max 2000 chars, optional)
- Reject unknown/extra fields in the request body — do not silently accept or store them
- Reject requests where `id` in the URL is not a valid integer/UUID format before hitting the DB
- Strip or reject HTML/script content in `title`/`description` (stored-XSS prevention) — specify whether this is done via encoding on output or rejection on input, and be consistent
- Date fields must be parsed with a strict format check (reject malformed strings like `"tomorrow"` or `"2026-13-45"`) before any timezone conversion is attempted

**Query safety**
- All database access uses better-sqlite3 **parameterized/prepared statements** — no string concatenation or template literals building SQL, even for internal fields
- State explicitly that dynamic `ORDER BY`/filter clauses (if any) must come from an allowlist of column names, never from raw user input

**Mass assignment / field allowlisting**
- `PUT /api/todos/[id]` only updates fields explicitly present in a defined allowlist (title, description, due_date, completed) — server-controlled fields like `id`, `created_at` must be ignored even if present in the request body
- Specify what happens if the client sends a field like `completed_at` or tries to change `created_at` — it's silently dropped, not an error, but also not applied

**Error handling**
- Define a standard error response shape (e.g. `{ error: { code, message } }`) used across all 5 endpoints
- Internal errors (DB failures, unexpected exceptions) return a generic 500 message to the client; the real error/stack is only logged server-side, never returned in the response body
- Validation errors return 400 with a field-level message (e.g. `{ error: { code: "INVALID_DUE_DATE", message: "Due date must be at least 1 minute in the future" } }`)
- Not-found (`GET/PUT/DELETE` on a missing id) returns 404, not 500 or an empty 200
- Define behavior for a DB write failure mid-request (e.g. delete cascade partially fails) — must not leave orphaned rows; specify whether it's wrapped in a transaction

**Rate limiting / abuse prevention**
- State the expected request-size limit for the request body (e.g. reject bodies over a defined KB limit before parsing)
- Note (even if enforcement lands with Feature 11 auth) that write endpoints (`POST`/`PUT`/`DELETE`) should be identified as needing per-session/per-IP rate limiting once authentication exists — flag this as a dependency rather than silently ignoring it

**HTTP semantics & idempotency**
- `DELETE` on an already-deleted or nonexistent id returns 404, not a 200/204 — deletes are not silently idempotent-success
- `PUT` on a nonexistent id returns 404, does not upsert/create
- Correct status codes throughout: 201 (create), 200 (read/update), 204 or 200 (delete — pick one and be consistent), 400 (validation), 404 (not found), 500 (server error)

**Timezone-specific guardrails**
- Specify exactly where timezone conversion happens (at the API boundary, using `lib/timezone.ts`) so it can't be bypassed by a client sending a pre-converted UTC value that fails the "future" check inconsistently
- Define behavior for a `due_date` with no timezone info in the string, vs. one with an explicit offset

**Logging & observability**
- Log validation failures and 500s with enough context to debug (endpoint, id, error type) but explicitly exclude full request bodies from logs if they could later contain sensitive fields (forward-looking, since auth/PII may be added later)
